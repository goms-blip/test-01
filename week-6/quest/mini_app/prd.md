
## 📝 유료 콘텐츠 잠금 해제 미니앱

## 1. 프로젝트 개요 (Project Overview)

* 프로젝트명: 간단한 베이킹 지식 상점
* 한 줄 정의: 결제를 통해 고품질 콘텐츠의 접근 권한을 구매하고 열람하는 마이크로 결제 플랫폼.
* 배경: 단순히 물건을 배송받는 커머스를 넘어, 디지털 콘텐츠의 '접근 권한(Access Right)'을 즉시 구매하고 소비하는 SaaS/콘텐츠 비즈니스 모델을 구현함.

------------------------------
## 2. 핵심 가치 제안 (Value Proposition)

* 사용자: 무료로 제공되는 일부 내용을 보고 가치를 판단하여, 필요한 정보를 저렴한 가격(단건 결제)에 즉시 획득.
* 운영자: 무분별한 콘텐츠 복제를 방지하고, 토스페이먼츠를 통한 안전한 수익화 실현.

------------------------------
## 3. 타겟 유저 (Target Audience)

* 특정 분야의 전문 지식(레시피, 템플릿, 노하우)을 유료로 구매할 의사가 있는 사용자.
* 복잡한 구독 서비스보다 필요한 정보만 골라 사고 싶은 실용주의적 소비자.

------------------------------
## 4. 핵심 기능 요구사항 (Key Features)
## 4.0. 사용자 계정 (Authentication)

* 회원가입: 이메일 + 비밀번호로 가입. 비밀번호는 서버에서 salt + scrypt 해시로 저장(평문 미저장).
* 로그인 / 로그아웃: 로그인 시 서버가 발급한 토큰을 클라이언트가 저장(`localStorage`)하고, 이후 모든 보호 API 호출 시 `Authorization: Bearer <token>` 헤더로 송신.
* 인증 기준:
  * 미인증 — 콘텐츠 목록·미리보기·인기 베스트 3까지 열람 가능.
  * 인증 필요 — 결제(intent/confirm), 마이 라이브러리(`/api/purchases`).
* 데모 계정: `demo@local / demo1234` (서버 부트 시 자동 시드) — 로그인 페이지에서 안내.
* 결제 위젯의 `customerKey`는 서버 발급 `userId`(`u_n`)를 사용하여 동일 사용자에 대해 안정적 매핑 보장.

## 4.1. 콘텐츠 브라우징 및 미리보기 (Discovery)

* 목록 화면: 전체 콘텐츠 리스트와 함께 가격 정보를 명확히 노출.
* 미리보기 UX: 본문의 상단 3줄만 공개하여 독자의 호기심 유발 및 가치 확인 기회 제공.
* 잠금 화면: 결제 전 본문 영역을 블러(Blur) 처리하거나 잠금 메시지 표시.

## 4.2. 토스페이먼츠 결제 시스템 (Payment)

* 결제 위젯: 토스페이먼츠 SDK를 활용한 직관적인 결제 UI 제공.
* 서버 승인 로직: 프론트엔드가 아닌 서버 API에서 결제 승인을 최종 확인하여 보안 강화.
* 구매 기록 저장: 결제 완료 시 purchases 테이블에 user_id와 content_id를 매핑하여 영구 소장 처리.

## 4.3. 접근 제어 및 열람 (Access Control)

* 권한 검증: 콘텐츠 상세 페이지 로드 시, 서버에서 해당 사용자의 구매 이력을 조회.
* 조건부 렌더링:
* 구매 이력 있음 → 전체 본문(body) 반환 및 표시.
   * 구매 이력 없음 → 본문 제외, 결제 유도 버튼 반환.
* 보안: 개발자 도구를 통한 우회를 방지하기 위해 서버에서 본문 데이터를 필터링하여 전송.

## 4.4. 마이페이지 (My Library)

* 접근 제한: 로그인한 사용자만 진입 가능. 미인증 시 로그인 페이지로 안내.
* 사용자 정보: 로그인된 이메일 표시 + 로그아웃 버튼.
* 구매 이력 페이지: 사용자가 구매한 모든 콘텐츠를 썸네일·제목·결제 일자·결제 금액과 함께 한눈에 확인.
* 재열람 기능: 한 번 구매한 콘텐츠는 추가 결제 없이 언제든 다시 열람 가능 — 다른 기기/브라우저에서도 같은 계정으로 로그인하면 동일한 라이브러리 노출.

## 5. 데이터 모델 (Data Model)

| 테이블명 | 주요 필드 |
|---|---|
| users | id(`u_n`), email, salt, password_hash(scrypt), created_at |
| sessions | token, user_id (인메모리 Map. 운영 시 Redis/JWT) |
| contents | id, title, preview(앞3줄), body(전체), price, thumbnail |
| purchases | id, user_id, content_id, paid_at, amount, toss_payment_key, toss_order_id |

------------------------------
## 6. 성공 지표 (Success Metrics)

* 결제 전환율: 콘텐츠 상세 페이지 방문자 대비 구매 완료 비중.
* 재구매율: 1회 이상 결제한 유저가 다른 콘텐츠를 추가로 구매하는 비율.
* 이탈률: 결제 화면에서 실제 결제 완료까지 도달하지 못하고 나가는 비율.

------------------------------
## 7. 중요 보안 포인트 

* 서버에서 데이터를 숨기는 것
* 프론트엔드에서 display: none으로 처리하면 유저가 금방 뚫을 수 있으니, API 응답 자체를 권한에 따라 다르게 설계했는지 확인
* **인증**: 비밀번호는 평문 저장 금지(salt + scrypt 해시). 토큰은 서버 발급 후 인메모리 sessions에서만 조회 가능.
* **결제 검증**: 클라이언트가 보낸 amount/userId를 그대로 신뢰하지 않음.
  * `amount` — 서버의 contents 가격과 대조 (`amount_mismatch`).
  * `userId` — body·query 무시, `Authorization` 헤더의 토큰에서 추출한 `req.userId`만 사용.
* **결제 승인**: 반드시 서버에서 TossPayments `/v1/payments/confirm` 호출로 검증. 클라이언트 성공 콜백만으로는 절대 구매 처리 금지.
